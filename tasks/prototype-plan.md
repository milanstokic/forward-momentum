# Implementation Plan: Extension A — Prototype Module

> **Extends** [`SPEC.md`](../SPEC.md) and the v1 plan in [`plan.md`](./plan.md). This is an
> **additive module**, not a rewrite. Source build spec: "Forward-Momentum — Extension A: Prototype
> Module" (extends Product Brief §8). Read alongside the v1 plan; don't duplicate its content.

## Overview

Add a **throwaway clickable prototype** capability to Forward-Momentum. On demand (from Gap analysis
/ Resolution), generate a self-contained **static** prototype from the current PRD + claims + gaps.
To be clickable, the prototype is **forced to choose a path** through targeted open **BLOCKING**
gaps; each forced choice is recorded in a manifest *and* surfaced in-UI as a provisional banner.
Teammates view it locally (open-in-browser + webview, **zero deploy**), leave **screen-anchored
reactions**, and those reactions feed back into the gap-analysis skill as **new or sharpened gaps**.
The prototype's job is to surface disagreement faster than prose — then be discarded.

The wedge (and the only success metric that matters): a reaction produces a **new gap that no text
source contained** — a requirement that only became visible once someone could click.

## Architecture Decisions

- **Throwaway-first.** `/prototype/` is scratch: regenerated on demand, never hand-edited, history
  ignored, no meaningful versioning. Treat it like a build artifact, not a deliverable. (§3, §4)
- **Static, no-network hard constraint.** Output is a single self-contained `index.html` (inline
  CSS/JS) or a static multi-file build with **no network calls, no backend, no env/secrets**; all
  data is inline **synthetic** mock data. This is an acceptance criterion *and* a non-goal guard
  (AC1, AC6). No deploy/CI/PR-preview/ephemeral-URL code is introduced by this module.
- **LLM-authored via the existing skill mechanism.** The prototype is generated by a Claude Code
  skill (`fm-prototype`) invoked via the `claude` CLI subprocess, exactly like `fm-extraction` /
  `fm-gap-analysis`. Keeping it a standalone skill means it is testable against
  `examples/sample-engagement/` without the UI (mirrors v1 Track A).
- **Reaction → gap is a *first-class input*, not a new pipeline.** We **extend the existing
  `fm-gap-analysis` skill** to ingest `/prototype/reactions.jsonl`, rather than building a parallel
  analyzer. One gap brain, more inputs. New gaps carry provenance `prototype@<screen-id>`.
- **Single source of truth = files on disk** (per v1). Manifest, reactions, and prototype output are
  all files the panels re-read on change. No divergent in-memory state.
- **On-demand, deadlock-breaking — not a pipeline stage.** The generate action takes **target gap
  ID(s)** and forces *those*. "Prototype everything" is discouraged to cap throwaway-UI noise. A
  soft "prototype reviewed" record may exist but **does not hard-block Handoff**. (§8)
- **Local viewing prefers a localhost static server** over `file://` (which breaks module/routing);
  `file://` is fallback only. (§5)

## Data Contracts (new — add to `src/model/prototype.ts`)

```typescript
// A path chosen through one open BLOCKING gap to make the prototype clickable.
export interface ProvisionalChoice {
  gapId: string;     // the blocking gap this choice forces, e.g. "gap-001"
  choice: string;    // the path taken, e.g. "guest checkout"
  rationale: string; // why this path (e.g. "call @06:30")
}

// The prototype manifest — what was forced, and how to render the banners.
export interface PrototypeManifest {
  generatedAt: string;            // ISO8601
  targetGapIds: string[];         // gap IDs this generation was asked to force
  choices: ProvisionalChoice[];   // one per forced blocking gap (may be [])
  screens: string[];              // screen-ids present in this prototype (for stale-anchor checks)
}

// A teammate reaction, anchored to a screen (optionally an element).
export interface Reaction {
  id: string;       // stable id
  author: string;   // identity string (single-user demo: config string)
  screen: string;   // screen-id the reaction anchors to
  element?: string; // optional element selector/label within the screen
  text: string;     // the comment
  ts: string;       // ISO8601
}

// Provenance anchor for downstream gap-analysis use:
export const protoAnchor = (screenId: string) => `prototype@${screenId}`;
```

Reactions are stored as JSON-lines in `/prototype/reactions.jsonl` (one `Reaction` per line).

### Phase D: The wedge (reaction → gap)

flagged `stale-anchor`, and still fed in.
**Acceptance criteria:**
- [ ] `fm-gap-analysis/SKILL.md` documents reactions as an input and the three resolution outcomes (sharpen / new / confirm-provisional), plus the interacting-choices note and `stale-anchor` handling.
- [ ] **AC4:** a captured reaction appears as an input to the next gap-analysis run, and any resulting gap is tagged provenance `prototype@<screen>`.
- [ ] **AC5:** a reaction can produce a **new** gap absent from all text sources (the worked example: guest-checkout prototype → "no post-purchase order access for guests" new gap).

**Verification:**

**Dependencies:** P4 (reactions exist), v1 T5

## Parallelization

- **P1 first** — it's the shared contract; everything else depends on it.
- **Two tracks after P1** (mirrors v1 Track A/B): the **skill track** (P2 → P5a) is standalone and
  testable against the sample without the UI; the **UI track** (P3 → P4) needs v1 panels. Run as two
  subagents; they converge at P5b/P6.
- **P5b and P6** are sequential and stay on the main thread (they wire shared panel + flow state).

## Risks and Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Generated prototype is too thin to provoke real reactions (the bet fails) | High | **Checkpoint A** exercises P2 on the sample before any viewing UI is built. Fail fast; tune the skill prompt or rethink. |
| A prototype sneaks in a network call / backend (breaks AC1, AC6) | High | Automated no-network grep assertion in P2's golden test; AC6 repo-wide grep guard in P6. Static-only is enforced by test, not just by prompt. |
| Reaction → gap produces only restatements, never a *new* gap (no wedge) | High | AC5 test asserts a gap present *with* the reaction is absent from a baseline run *without* it — proves novelty, not just presence. |
| Module blocked on unbuilt v1 panels (T8/T9) | Med | Skill track (P1/P2/P5a) needs no UI and proceeds now; UI track falls back to command-palette entry points until panels land. |
| `/prototype/` scratch accidentally committed / hand-edited | Low | Gitignore `prototype/` output (keep only `.gitkeep` + fixture `reactions.jsonl`); document "never hand-edit" in the skill. |
| Real client data embedded in mock data | Low (local viewing) but required | Skill instruction + test that mock data is synthetic; no source `quote` text copied verbatim into shippable mock fields. |

## Open Questions (non-blocking; resolve during build)

1. **Screen-id scheme** — how does the generator name screens so reactions anchor stably across
   regenerations (semantic ids like `confirmation-screen` vs positional)? Semantic ids preferred for
   stable anchoring; settle in P2. (Drives stale-anchor behavior in P5a.)
2. **Reaction author identity** — reuse v1's single-user config string; confirm where that string
   lives once T7/T9 exist.
3. **Gitignore vs fixture** — the sample needs a committed `reactions.jsonl` fixture for the AC5 test,
   but generated `prototype/` output should be ignored. Carve the fixture out of the ignore rule.
4. **Manifest screens source of truth** — does the generator emit `screens[]`, or do we derive it by
   scanning the HTML? Generator-emitted is more reliable; confirm in P2.
